# Onboarding: как работать с проектом, ботами и сервером

Привет! Это краткий гайд по проекту `tg-nav-bot`. Здесь два Telegram-бота,
собранные из **одного кода**, но запускаемые как **разные контейнеры** с
разными `.env` и разными `docker-compose`-файлами.

---

## 1. Что это за проект

| Бот | Что делает | Контейнер | Compose-файл | Сервис |
|-----|------------|-----------|--------------|--------|
| **Навигационный (нав-бот)** | Прод-бот: навигация по главам веб-новеллы, поиск, оглавление | `rqm-nav` (прод) | `docker-compose.yml` | `rqm-nav` |
| **ИИ-аватары** | Тестовый бот: персонажи новеллы отыгрывают себя в групповом чате (сейчас в фокусе Ютия) | проект `rqm-test` | `docker-compose.test.yml` | `rqm-test` |

Боты полностью изолированы: **разные BOT_TOKEN, разные тома, разные БД**
(`rqm.db` у нав-бота, отдельный том `rqm-test-data` у ИИ-бота). Два поллера на
одном токене ломают друг друга — токены ОБЯЗАТЕЛЬНО разные.

---

## 2. Ветки и деплой

4 ветки, по две на каждого бота:

| Ветка | Бот | Назначение | Авто-деплой |
|-------|-----|-----------|-------------|
| `master` | нав | **чистовая** (прод) | ✅ push → деплой нав-бота |
| `nav-test` | нав | тестовая (пробуем изменения) | ❌ только тесты в CI |
| `ai-prod` | ИИ | **чистовая** (прод ИИ-бота) | ✅ push → деплой ИИ-бота |
| `ai-test` | ИИ | тестовая (тут идёт вся разработка ИИ) | ❌ только тесты в CI |

**Правило работы:** изменения сначала в тестовую ветку → проверяем (CI + руками
на тестовом контейнере) → когда ок, переносим в чистовую (merge/fast-forward) →
авто-деплой подхватывает.

Авто-деплой описан в `.github/workflows/deploy.yml`:
- на любой push/PR прогоняются тесты + сборка Docker-образа;
- push в `master` → SSH-деплой нав-бота, push в `ai-prod` → SSH-деплой ИИ-бота.

> ⚠️ Чтобы авто-деплой заработал, в **Settings → Secrets and variables →
> Actions** репозитория должны быть добавлены секреты (см. §6). Пока их нет —
> джоб деплоя просто пропускается (CI остаётся зелёным).

## 3. Рабочий цикл (типовой)

```bash
# 1. начни от тестовой ветки нужного бота
git checkout ai-test && git pull

# 2. поправь код / карточку персонажа, прогони тесты локально
python -m pytest -q

# 3. закоммить и запушь в тестовую
git add -A && git commit -m "..." && git push

# 4. убедись, что CI зелёный и тестовый бот ведёт себя ок,
#    затем перенеси в чистовую — это запустит авто-деплой
git checkout ai-prod && git merge --ff-only ai-test && git push
```

Для нав-бота то же самое с парой `nav-test` → `master`.

## 4. Сервер

Деплой-модель: на сервере лежат **два клона репозитория** (по одному на бота).
Git обновляет только КОД; `.env` и Docker-тома живут на сервере и в git не
попадают.

Примерная раскладка (точные пути — у владельца):
```
/opt/rqm-bot/        # нав-бот: ветка master,  docker-compose.yml
/opt/rqm-bot-test/   # ИИ-бот:  ветка ai-prod,  docker-compose.test.yml
```

### Ручной деплой (если нужно без CI)
Скрипт `scripts/update_server.sh` тянет ветку, пересобирает контейнер,
прогоняет smoke-проверку и **сам откатывается** при провале.

```bash
# нав-бот (дефолтное поведение)
cd /opt/rqm-bot && DEPLOY_BRANCH=master ./scripts/update_server.sh

# ИИ-бот (свой compose/проект/сервис)
cd /opt/rqm-bot-test && \
  DEPLOY_BRANCH=ai-prod COMPOSE_FILE=docker-compose.test.yml \
  COMPOSE_PROJECT=rqm-test DEPLOY_SERVICE=rqm-test \
  ./scripts/update_server.sh
```

### Полезные команды на сервере
```bash
# логи
docker compose logs -f                                   # нав-бот
docker compose -f docker-compose.test.yml -p rqm-test logs -f   # ИИ-бот

# статус / перезапуск
docker compose ps
docker compose -f docker-compose.test.yml -p rqm-test restart

# остановить ИИ-бот (том сохраняется)
docker compose -f docker-compose.test.yml -p rqm-test down
```

`.env` на сервере держи с правами `chmod 600` (скрипт деплоя предупредит, если нет).

## 5. Локальная разработка

```bash
# зависимости
python -m pip install -r requirements.txt pytest

# тесты (должны быть зелёными перед пушем)
python -m pytest -q

# оффлайн-аудит маршрутизации/промпта Ютии (без обращения к API)
python scripts/yutia_eval_offline.py
```

Для локального запуска самого бота нужен `.env` (скопируй из `.env.example` /
`.env.test.example` и впиши свои `BOT_TOKEN` и `AI_API_KEY` — это **Groq**-ключ).
Без `AI_API_KEY` ИИ-функция выключается, бот работает как обычный нав-бот.

### Карточки персонажей
Лежат в `personas/*.json`. Сейчас доведена до ума только **Ютия**
(`personas/yutia.json`). Структура богатой карточки v2: `identity`, `registers`
(режимы речи), `routing` (как сообщение → регистр/накал), `relationships`,
`knowledge_boundaries`, `style_notes` (жёсткие правила формулировок),
`example_dialogues`. Схема — `personas/profile.schema.json`.

## 6. Структура проекта

```
app/
  ai/                 # движок ИИ-персонажей (v2)
    engine.py         #   очередь, генерация ответа, ретраи, трейсы
    planner.py        #   ReplyPlan: интент/регистр/накал/сущности по сообщению
    prompting.py      #   динамическая сборка промпта под сообщение
    personas.py       #   загрузка карточек, выбор примеров, правила письма
    knowledge.py      #   подбор фактов из базы знаний (RAG по главам/сценам)
    quality.py        #   валидация ответа (роль/формат) перед отправкой
    store.py          #   SQLite: память, отношения, трейсы, KB (ai.db)
    kb_builder.py     #   фоновая сборка выжимок глав (gpt-oss)
    models.py         #   датаклассы (ReplyPlan, KnowledgeBundle, ...)
  bot.py              # хэндлеры Telegram, команды /ai ...
  main.py, entrypoint.py, api.py, config.py, db.py   # каркас нав-бота
  parser.py, render.py, telegraph.py, registry.py    # навигация/контент
  smoke.py            # smoke-проверка (дергается деплой-скриптом)
personas/             # карточки персонажей + лор + сценарии оценки
  yutia.json, lore.md, profile.schema.json
  eval/yutia_scenarios.json
scripts/
  update_server.sh    # деплой на сервере (параметризован под оба бота)
  yutia_eval_offline.py  # оффлайн-аудит Ютии
docker-compose.yml         # нав-бот (прод)
docker-compose.test.yml    # ИИ-бот (изолированный тестовый стенд)
Dockerfile
.github/workflows/deploy.yml  # CI + авто-деплой
DEPLOY.md, TESTING.md, README.md  # доп. документация
```

ИИ-бот пишет в свою БД `ai.db` (память/отношения/трейсы), отдельную от
навигационной `rqm.db`.

## 7. Что нужно настроить для полного доступа

1. **Запись в репо** (push): владелец добавляет твой GitHub-ник в
   Settings → Collaborators. Без этого ты можешь читать/форкать (репо публичный),
   но не пушить в ветки.
2. **Сервер**: владелец добавляет твой публичный SSH-ключ в `~/.ssh/authorized_keys`
   нужного пользователя на сервере. Пришли ему свой `id_ed25519.pub`.
3. **Авто-деплой** (GitHub Secrets, добавляет владелец):
   - `DEPLOY_SSH_HOST` — IP/домен сервера
   - `DEPLOY_SSH_USER` — пользователь для деплоя
   - `DEPLOY_SSH_KEY` — приватный SSH-ключ деплой-пользователя
   - `DEPLOY_SSH_PORT` — порт SSH (если не 22)
   - `NAV_DIR` — путь к клону нав-бота (напр. `/opt/rqm-bot`)
   - `AI_DIR` — путь к клону ИИ-бота (напр. `/opt/rqm-bot-test`)

## 8. Важные нюансы

- **Бесплатный тариф Groq**: главное ограничение — rate-limit. Не гоняй лишних
  генераций; чат-модель — только Llama (gpt-oss ломает образ персонажа).
- **Прод vs тест**: не путай контейнеры. `master`/`docker-compose.yml` — это
  живой нав-бот; меняй его осознанно.
- **Перед пушем в чистовую ветку всегда прогоняй тесты** — push туда = деплой.
