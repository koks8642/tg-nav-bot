# Задача: автоматическая навигация для Telegram-канала переводов (RQM)

## Что строим

Система автоматической навигации по Telegram-каналу переводческой команды RQM. Автор постит главы новелл (и арты/мемы/заметки), бот за всем этим следит и поддерживает навигацию в актуальном состоянии **без ручного труда**. Сами главы читаются на Telegraph (туда автор их и публикует) — мы строим слой навигации поверх.

**Навигация существует в ДВУХ представлениях поверх одной общей БД:**

1. **Telegraph-страницы** — статичная «витрина». Главная страница закреплена в канале, выглядит как у эталона (https://telegra.ph/Navigaciya-po-novelle-Tenevoj-Rab-10-02), но по всему каналу и всем проектам. Стандартная навигация переходами по ссылкам. Ссылку можно кинуть другу, открывается у всех.

2. **Telegram Mini App** — интерактивное представление с **умным быстрым поиском** глав (по номеру/названию/проекту) и просмотром навигации и активностей команды. Открывается кнопкой из закреплённого поста, работает внутри Telegram.

Оба представления — это просто разные «виды» одних и тех же данных. Бот их синхронно обновляет при каждом новом посте. Дублирования логики нет: бот пишет в БД, Telegraph-рендерер и Mini App API читают из неё.

```
┌─────────────── Канал (автор постит) ───────────────┐
                         │
                    БОТ (watcher)
              хэштег-центричный разбор
                         │
                   БД (SQLite) ← единственный источник истины
                    /          \
        Telegraph-рендерер    Mini App API (HTTP)
              │                      │
        статичные страницы      Mini App (поиск/навигация)
        (закреплены в канале)   (кнопка в закрепе)
```

Канал **приватный**. Внутренний ID канала: `3131929652` (ссылки на посты строятся как `https://t.me/c/3131929652/<message_id>`). Точный `chat_id` для Bot API нужно взять из апдейтов (формат `-1003131929652`) и вынести в конфиг.

## КРИТИЧЕСКИ ВАЖНЫЙ КОНТЕКСТ (из анализа реального экспорта канала, 194 поста)

Прежде чем проектировать, осознай реальность данных. В папке `./ChatExport/` лежит выгрузка канала (`messages.html` + медиа). Изучи её. Ключевые факты:

1. **Хэштегов в канале сейчас НЕТ ни одного.** Систему строим хэштег-центричной (см. раздел «Архитектура»), но это значит: (а) автор начнёт тегать НОВЫЕ посты, (б) вся существующая история (360+ глав) не имеет тегов и заливается одноразовым структурным бэкафиллом из экспорта. Не рассчитывай, что история размечена.

2. **Один пост = пачка глав (1–8 штук), а не одна запись.** Типичный формат:
   ```
   🎁 Новые главы 🎁
   Новелла: "Стал Покровителем Злодеев"
   Главы 282-289 «Приём»
   Глава 282   (ссылка → telegra.ph/...-Glava-282-...)
   Глава 283   (ссылка → telegra.ph/...-Glava-283-...)
   ...
   ```
   Каждая строка «Глава N» — отдельная гиперссылка на отдельную страницу Telegraph. Парсить надо КАЖДУЮ ссылку как отдельную главу.

   Встречается и старый формат (1 пост = 1 глава): `Ошибочно Приняли За Величайшего Гения, Глава 5 (Подобранная ③)` + одна ссылка.

3. **Канал приватный** → ссылки на посты только в формате `https://t.me/c/3131929652/<id>`. Публичный username отсутствует.

4. **Telegraph живёт на двух доменах:** `telegra.ph` И `graph.org`. Распознавай оба как «ссылку на главу».

5. **Названия проектов грязные** — опечатки («Величайшего Гения» vs «Гений»), переименования по ходу. Нельзя матчить проект по сырой строке названия. Нужен реестр проектов с каноническим именем + список алиасов/паттернов. Хэштег проекта (когда появится) — самый надёжный идентификатор.

6. **Есть дубли и ошибки в данных** (одна глава запощена дважды, у некоторых битый номер). Нужна дедупликация по `(project, chapter_number)` с правилом «оставляем последнюю/исправленную версию».

7. **Внешние ссылки уже разбросаны по истории:** RanobeLib (`ranobelib.me/ru/book/...`), MangaLib (`mangalib.me/ru/manga/...`), Boosty (`boosty.to/becamerqm`). Их надо привязывать к проектам.

8. **Реальные проекты в истории** (неполный список — собери актуальный из экспорта):
   - «Стал Покровителем Злодеев» (RanobeLib book 214126) — ~300 глав
   - «Ошибочно Приняли За Величайшего Гения» (RanobeLib 246670) — ~57 глав
   - «Отличный урожай, мой повелитель!» (RanobeLib 231687, MangaLib 172893)
   - «Как покорить Башню Ханамджи» / «Башня» (RanobeLib 238087)
   - «Теневой Духовный Дракон»
   - и др.

## Архитектура: ХЭШТЕГ-ЦЕНТРИЧНАЯ система

Решение по продукту принято: навигация строится **на хэштегах**. Хэштег на посте — единственный источник истины о том, к какому проекту/категории относится пост. Бот не угадывает проект по тексту названия (из-за опечаток и переименований это ненадёжно) — он читает хэштег.

**Правило «один хэштег на пост».**
Автор ставит на пост ровно один хэштег:
- `#покровитель`, `#гений`, `#урожай`, `#башня`, … — хэштег ПРОЕКТА. Весь пост относится к этому проекту.
- `#арты`, `#мемы`, `#заметки_переводчика`, `#анонсы` — хэштег КАТЕГОРИИ (нечаптерный контент).

Сопоставление `хэштег → проект` и `хэштег → категория` хранится в БД (`hashtag_map`) и настраивается через админку. Один проект может иметь несколько хэштегов-синонимов.

**Извлечение глав внутри тегированного поста.**
Хэштег говорит «это проект X». Дальше бот разбирает тело поста и достаёт ВСЕ главы (в одном посте их 1–8): для каждой строки `Глава N` со ссылкой → `(номер N, название_арки, telegraph_url)`. Плюс ссылка на сам пост (`t.me/c/3131929652/<msg_id>`). То есть тег = принадлежность, структурный парсинг тела = список глав. Поддерживай оба домена `telegra.ph` и `graph.org`.

**Новый неизвестный хэштег → новый раздел.**
Если приходит пост с хэштегом, которого нет в `hashtag_map`, бот:
- создаёт новую категорию/раздел с этим именем автоматически (как просил заказчик),
- уведомляет владельца в админке, чтобы тот при желании переименовал/привязал её к проекту.

**Бэкафилл истории (одноразовый, структурный — у старых постов тегов нет).**
360+ существующих глав опубликованы БЕЗ хэштегов. Скрипт-импортёр парсит `./ChatExport/messages.html` и наполняет БД всей историей. Поскольку тегов в истории нет, бэкафилл определяет проект структурно — по строке `Новелла: "..."` / префиксу `<Название>, Глава N`, сопоставляя с реестром проектов через алиасы (`project_aliases`). Извлечение глав — тем же парсером тела, что и в live. Бэкафилл **идемпотентен** (повторный запуск не плодит дубли). После него навигация уже полная; дальше всё живёт на хэштегах.

> Важно: структурный матчинг проекта используется ТОЛЬКО в бэкафилле истории. В live-режиме проект определяется исключительно хэштегом.

## Представление 1 — Telegraph-страницы (как у эталона, многоуровневые)

```
ГЛАВНАЯ Telegraph-страница (закреплена в канале)
├── Проекты
│   ├── Стал Покровителем Злодеев  → страница проекта
│   │   ├── Главы (сгруппированы по аркам/томам, каждая = «Читать в Telegraph» + «Пост в канале»)
│   │   ├── Арты
│   │   ├── Заметки переводчика
│   │   └── Читать на: RanobeLib / MangaLib / Boosty   ← внешние ссылки ВНУТРИ проекта
│   ├── Ошибочно Приняли За Величайшего Гения → ...
│   └── ...
├── Арты (агрегат по всем проектам)
├── Мемы
├── Уголок переводчика (агрегат заметок)
└── Анонсы
```

Требования к навигации:
- **Многоуровневость, но быстрый доступ.** Это навигация — кликов до цели минимум. Главная → раздел → список. Каждый уровень — отдельная Telegraph-страница со своим стабильным path; верхние уровни ссылаются на нижние.
- **Главы внутри проекта** группируются по аркам (название арки парсится из текста/слага: «Арена», «Ведьма», …) или по томам. Сортировка по номеру главы по возрастанию.
- **Двойная ссылка на главу:** «Читать в Telegraph» (прямая на страницу главы) и «Пост в канале» (на TG-пост, где глава анонсирована).
- **Кросс-категоризация:** пост с несколькими тегами (например `#покровитель` + `#арты`) попадает и в «Арты» проекта, и в глобальные «Арты». Один источник — несколько мест в навигации.
- **Внешние ссылки (RanobeLib/MangaLib/Boosty) лежат ВНУТРИ карточки проекта**, не на верхнем уровне. Обновляются автоматически, если автор постит их с тегом проекта; плюс редактируются через админку.
- Оформление (эмодзи-иконки разделов, заголовки, разделители) — в стиле эталона.

## Представление 2 — Telegram Mini App (интерактив + умный поиск)

Mini App (Telegram WebApp) — лёгкое веб-представление той же БД. Должно быть **быстрым и приятным** (это явное требование заказчика), не тяжёлым SPA. Цель — за пару тапов найти нужную главу или раздел.

**Технически проще, чем кажется:** это статика (HTML + минимальный ванильный JS или лёгкий фреймворк) + Telegram WebApp SDK (`telegram-web-app.js`, одна строчка) + тонкий HTTP/JSON API к той же БД. Никакого тяжёлого фронтенда. Открывается через кнопку `web_app` в закреплённом посте / в меню бота.

**Экраны:**
- **Главная:** строка поиска сверху + плитки/список разделов (Проекты, Арты, Мемы, Уголок переводчика, Анонсы) и список проектов.
- **Умный поиск (ключевая фича):** мгновенный (debounce, поиск на лету) по номеру главы, названию арки, названию проекта. Примеры запросов: «покровитель 245», «245», «арена», «башня». Результаты — кликабельные, ведут на Telegraph-главу. Поиск работает по всей БД, мгновенно.
- **Страница проекта:** список глав (виртуализированный/пагинированный скролл — выдерживает 300+ глав без тормозов), группировка по аркам, фильтр/поиск внутри проекта, блок внешних площадок (RanobeLib/MangaLib/Boosty), разделы Арты/Заметки проекта.
- **Глава:** «Читать в Telegraph», «Пост в канале».
- **Активности команды:** лента последних публикаций (что и когда вышло) — витрина деятельности команды.

**API (бэкенд Mini App).** Тонкий асинхронный HTTP-слой (тот же процесс или рядом):
- **Публичное чтение:** `/api/sections`, `/api/projects`, `/api/project/{id}`, `/api/search?q=`, `/api/chapter/{id}` — отдаёт JSON, только чтение.
- **Админ (write):** `/api/admin/*` — CRUD по всем сущностям (см. раздел «Система управления»), доступно только владельцам. Каждый запрос проверяет подпись Telegram `initData` и whitelist `OWNER_USER_IDS` на сервере.
- Валидация `initData` (подпись WebApp) на всех эндпоинтах. CORS под домен Mini App.

**Синхронность с Telegraph.** Mini App и Telegraph читают одну БД — они всегда консистентны автоматически. Никакой отдельной пересборки для Mini App не нужно: данные обновил бот → и страницы, и API сразу актуальны.

## Поток работы бота (runtime)

```
Новый/изменённый пост в канале (Bot API, channel_post / edited_channel_post)
  → есть ли хэштег? нет → пост служебный, игнор (в навигацию не идёт)
  → есть → ищем хэштег в hashtag_map:
        проект?   → разбираем тело, извлекаем все главы[], апсертим как главы проекта
        категория? → создаём item (арт/мем/заметка/анонс) со ссылкой и постом
        неизвестный → создаём новый раздел автоматически + уведомляем владельца
  → попутно ловим внешние ссылки (ranobelib/mangalib/boosty) и привязываем к проекту тега
  → upsert в БД (дедуп по (project, chapter_number); по message_id — для апдейтов)
  → постановка в очередь пересборки затронутых Telegraph-страниц
  → воркер очереди пересобирает страницы и вызывает Telegraph editPage
```
- Редактирование поста (`edited_channel_post`) → находим записи по `message_id`, обновляем, пересобираем.
- Удаление: Bot API не шлёт уведомления об удалении постов в канале — предусмотри ручную команду в админке «удалить запись/пересканировать» и периодическую сверку.

## Модель данных (SQLite, через aiosqlite)

```
projects(id, canonical_name, slug, telegraph_path, ranobelib_url, mangalib_url, boosty_url, sort_order, created_at)
project_aliases(id, project_id, alias_pattern)        -- строки/regex для матчинга грязных названий
hashtag_map(hashtag, kind, target_id)                 -- kind: 'project' | 'category'; target_id → project/section
sections(id, name, slug, emoji, telegraph_path, sort_order)   -- глобальные разделы (Арты/Мемы/Заметки/Анонсы)
posts(id, message_id, tg_url, date, raw_text, kind)
chapters(id, project_id, number, arc, title, telegraph_url, post_id, UNIQUE(project_id, number))
items(id, section_id, project_id, title, url, post_id, date)  -- арты/мемы/заметки (нечаптерный контент)
external_links(id, project_id, platform, url, title)
telegraph_pages(path, access_token, title, last_built_at)
build_queue(id, page_path, enqueued_at, status)
event_log(id, ts, level, message)
```

Эта же БД обслуживает оба представления: Telegraph-рендерер и read-only API Mini App читают из неё. Индексы под поиск Mini App (по `chapters.number`, `chapters.arc`, `projects.canonical_name`).

## Надёжность (это требование заказчика: «работать всегда, надёжно, не лажать»)

- **Идемпотентность везде.** Повторная обработка одного поста / повторный бэкафилл не создают дублей и не ломают навигацию.
- **Очередь пересборки** — страницы не пересобираются параллельно и не гонятся; апдейты дебаунсятся (пачка постов подряд → одна пересборка на страницу).
- **Ретраи Telegraph API** с экспоненциальным backoff; rate-limit Telegraph учитывается. Любая ошибка не роняет процесс — логируется в `event_log`.
- **Восстановление после рестарта** полностью из БД. Никакого состояния только в памяти.
- **Стартовая досборка:** при запуске бот подтягивает пропущенные за время простоя апдейты (Bot API даёт апдейты за ~24ч) и при необходимости предлагает ресканирование.
- **Бэкап БД** перед каждой массовой операцией (бэкафилл, ребилд).
- **Никогда не публиковать пустую/битую страницу** поверх рабочей: собрать новый контент, проверить, потом editPage.
- **Структурное логирование** + команда `/health` в админке (статус очереди, последние ошибки, кол-во проектов/глав).

## Система управления и редактирования (админка) — ВАЖНЫЙ блок

Заказчик прямо требует **удобную систему управления и редактирования всего**. Это не второстепенная функция, а полноценный инструмент: владелец должен мочь поправить/настроить ЛЮБУЮ сущность навигации руками, быстро и без программиста. Считай это полноценной CRUD-панелью над всей БД, с упором на удобство.

**Где реализовать.** Базовый уровень — inline-меню в чате с ботом (кнопки + пошаговые диалоги), доступно только владельцам (whitelist `OWNER_USER_IDS`). Плюс **админ-режим внутри Mini App** — те же действия, но с нормальным UI (формы, списки с поиском, drag-сортировка), доступен только владельцам (проверка по `initData`). Mini App-админка — основной удобный инструмент, бот-меню — быстрый доступ с телефона. Обе работают через один защищённый write-API.

**Что должно редактироваться (полный CRUD, без правки кода и редеплоя):**

- **Проекты:** создать / переименовать (каноническое имя) / задать эмодзи и обложку / привязать хэштег(и) / задать ссылки RanobeLib·MangaLib·Boosty / изменить порядок / скрыть·показать / удалить.
- **Алиасы проектов:** добавлять/править паттерны для матчинга «грязных» названий в бэкафилле и логах.
- **Разделы и категории:** создать / переименовать / эмодзи / порядок / удалить. Управление автосозданными (из неизвестных хэштегов) разделами — подтвердить, переименовать, слить с существующим, привязать к проекту.
- **Хэштеги (`hashtag_map`):** добавить/изменить/удалить связку `#тег → проект` или `#тег → категория`, увидеть все известные теги и их назначение.
- **Главы:** найти (поиск по проекту/номеру) → отредактировать номер, название арки, заголовок, Telegraph-URL; перепривязать к другому проекту; перенести в другую арку; удалить; восстановить.
- **Айтемы (арты/мемы/заметки/анонсы):** редактировать заголовок, ссылку, раздел, проект; удалить.
- **Арки/тома:** переименовать, изменить порядок, объединить, разбить.
- **Внешние ссылки:** добавить/править/удалить вручную (на случай, если автор не запостил их тегом).

**Операционные инструменты:**
- **Предпросмотр перед публикацией:** показать, как будет выглядеть страница/раздел до пересборки Telegraph.
- **Ручные действия:** «пересобрать страницу X», «полный ребилд всех страниц», «запустить/повторить бэкафилл из экспорта», «пересканировать пост по ссылке/ID».
- **Очередь дубликатов и конфликтов:** список подозрительных записей (дубли номеров, главы без проекта, неизвестные хэштеги, посты, которые бот не смог распознать) — с кнопками «исправить/привязать/игнорировать». Бот сам складывает сюда всё, в чём не уверен, вместо тихого пропуска.
- **Лог событий и health:** последние ошибки, статус очереди пересборки, счётчики (проектов/глав/айтемов), время последней успешной сборки. Команда `/health`.
- **Уведомления владельцу:** при неизвестном хэштеге, нераспознанном посте, ошибке Telegraph API — пуш в чат с быстрыми кнопками действий.
- **Аудит изменений:** лог ручных правок (кто, что, когда) — чтобы откатить при ошибке.
- **Безопасность:** все write-операции только для владельцев; Mini App-админка проверяет подпись `initData` и whitelist на сервере (не доверять клиенту).

## Стек и хостинг

- **Бэкенд:** Python 3.11+, `python-telegram-bot` v20+ (async), `aiosqlite`, `httpx`/`aiohttp` для Telegraph.
- **API Mini App:** лёгкий async-фреймворк (`aiohttp`/`FastAPI`+`uvicorn`) — read-only JSON-эндпоинты к той же БД. Может жить в том же процессе, что и бот, или рядом.
- **Фронтенд Mini App:** статика — HTML + ванильный JS (или лёгкий Preact/Alpine, без тяжёлого билда) + `telegram-web-app.js`. Цель — мгновенная загрузка. Раздаётся статикой (тем же сервером или Railway static).
- Telegraph API: `createAccount` → `createPage`/`editPage`. Свой тонкий клиент (готовые либы часто заброшены) с ретраями.
- Конфиг через переменные окружения: `BOT_TOKEN`, `CHANNEL_CHAT_ID`, `OWNER_USER_IDS`, `TELEGRAPH_TOKEN` (создать скриптом сетапа), `TELEGRAPH_AUTHOR`, `WEBAPP_URL`, путь к БД.
- Хостинг — **Railway.app** (24/7). БД SQLite на постоянном volume Railway (важно: эфемерная ФС теряет данные при редеплое — обязательно persistent volume). Деплой из GitHub-репозитория. Mini App требует HTTPS-домен — Railway даёт его из коробки.
- Polling (не webhook) для бота — проще и надёжнее для приватного канала на Railway.

## Обязательная последовательность реализации

1. **Парсер экспорта + реестр проектов.** Разобрать `./ChatExport/messages.html`, вывести список проектов, глав, внешних ссылок. Это и тест парсера, и источник бэкафилла. Покрыть юнит-тестами (форматы постов, извлечение номера/арки/URL, дедуп). **Показать результат и дождаться подтверждения, прежде чем идти дальше.**
2. **Модель БД + слой данных.**
3. **Telegraph-клиент** (создание/обновление страниц, сборка node-формата из данных) + рендер структуры по эталону.
4. **Бэкафилл** из экспорта (идемпотентный) → первая полная сборка всех Telegraph-страниц.
5. **Live-бот** (channel_post/edited_channel_post) на хэштег-пайплайне + очередь пересборки.
6. **API Mini App** (чтение + защищённые admin-write эндпоинты, валидация initData) — на той же БД.
7. **Фронтенд Mini App** (главная, умный поиск, страница проекта, глава, лента активностей).
8. **Система управления (админка):** bot-меню + админ-режим в Mini App — полный CRUD над всеми сущностями, очередь конфликтов, ручные действия, аудит. Это полноценный блок, а не заглушка.
9. **Деплой на Railway** + persistent volume + сетап Telegraph-токена + домен Mini App + закрепить главную Telegraph-страницу и кнопку Mini App в канале.

## Definition of Done

- Бэкафилл воспроизводит полную навигацию по всем существующим главам обоих основных проектов из экспорта, без дублей, с корректной сортировкой и арками.
- Новый пост с одним хэштегом проекта автоматически распознаётся, все главы из тела (до 8) попадают в нужные страницы в течение секунд, обе ссылки (Telegraph + пост) рабочие.
- Пост с хэштегом категории (`#арты`/`#мемы`/…) добавляет запись в соответствующий раздел.
- Неизвестный хэштег автоматически создаёт новый раздел и шлёт уведомление владельцу.
- Редактирование поста корректно обновляет навигацию.
- Внешние ссылки (RanobeLib/MangaLib/Boosty) видны внутри карточек проектов.
- Падение/рестарт процесса не теряет данные и не ломает опубликованные страницы.
- Админка даёт полный CRUD над всеми сущностями (проекты, разделы, хэштеги, главы, айтемы, арки, внешние ссылки), очередь конфликтов с ручным разбором, ручную пересборку/бэкафилл и аудит правок — всё без правки кода и редеплоя, через bot-меню и админ-режим Mini App.
- Оформление Telegraph-страниц соответствует стилю эталона.
- **Mini App** открывается из закрепа, грузится мгновенно; умный поиск находит главу по номеру/арке/проекту на лету; страница проекта держит 300+ глав без тормозов; кнопки «Читать в Telegraph»/«Пост в канале» рабочие.
- **Telegraph и Mini App консистентны** — показывают одни и те же данные, оба обновляются после нового поста без ручных действий.
